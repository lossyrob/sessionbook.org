# Public Set Pages

## Overview

This work turns public sets and tunes into linkable detail pages instead of browse-only rows. The public catalog now supports direct routes for individual sets and tunes, preserves the existing `/tunes` expandable browse experience, and adds bidirectional links so visitors can move naturally between a set and the tunes it contains.

The implementation stays inside the Release 1 public catalog boundary. It does not change seed data, search, auth, or owner-only flows.

## Architecture and Design

### High-Level Architecture

The change has three cooperating layers:

1. The Release 1 repository now exposes public-by-slug lookups for tunes and sets.
2. Public detail routes at `/sets/[slug]` and `/tunes/[slug]` render those records and provide route metadata for shareable titles and descriptions.
3. Existing browse surfaces (`/sets` and `/tunes`) now link into the detail routes without replacing their current interaction model.

### Design Decisions

- **Additive repository shape**: `PublicTuneView` keeps `setNames` for compatibility while adding `setMemberships` with slugs. That avoided breaking existing consumers while the detail-route work was added.
- **Minimal presentation reuse**: The new detail pages reuse existing classes such as `eyebrow`, `lead`, `callout`, `set-row`, and `tune-chart` instead of introducing a new styling system for this issue.
- **Preserve tune-list expansion**: The `/tunes` page still expands charts on row click. Links inside the row stop event propagation so navigation does not also toggle the chart.
- **Basic shareability via metadata**: Both detail routes export `generateMetadata` so browser tabs and previews reflect the current tune or set instead of generic site titles.

### Integration Points

- **`src/lib/release-1/repository.ts`** adds `getPublicTuneBySlug()` and `getPublicSetBySlug()` and enriches view data with tune/set slugs needed for links.
- **`src/app/sets/page.tsx`** and **`src/components/tune-list.tsx`** now serve as entry points into the detail routes.
- **`src/app/sets/[slug]/page.tsx`** and **`src/app/tunes/[slug]/page.tsx`** use `loadRelease1Repository()` and `notFound()` so they stay aligned with the rest of the app's data-loading and missing-route behavior.

## User Guide

### Prerequisites

- A running SessionBook app with the Release 1 catalog available from fixtures or Postgres.
- Any public set or tune slug generated by the existing seed data.

### Basic Usage

1. Open `/sets` to browse the public set index.
2. Click a set title to open its detail page.
3. Click any tune in that set to open the tune detail page.
4. From a tune page, use the linked set list to jump back to related public sets.
5. Open `/tunes` to browse the expandable list view; click a row to expand the chart or click the linked tune/set names to navigate instead.

### Advanced Usage

- Direct-link to a tune with `/tunes/<slug>` when sharing a chart-focused page.
- Direct-link to a set with `/sets/<slug>` when sharing a sequence of tunes for a session or gig context.
- Unknown tune or set slugs fall through to the shared not-found experience rather than rendering partial data.

## API Reference

### Key Components

- **`getPublicTuneBySlug(slug)`**: Returns a `PublicTuneView` with aliases, chart data, `setNames`, and slug-bearing `setMemberships`.
- **`getPublicSetBySlug(slug)`**: Returns a `PublicSetView` with ordered entries that include `tuneSlug` for navigation.
- **`TuneList`**: Continues to power the `/tunes` browse surface while now linking to detail pages and suppressing row-toggle side effects on link clicks.
- **Detail route pages**: `src/app/sets/[slug]/page.tsx` and `src/app/tunes/[slug]/page.tsx` render server-side pages backed by the repository lookups.

### Configuration Options

- The detail pages follow the app's existing `dynamic = "force-dynamic"` convention.
- Metadata is derived from repository content:
  - Set pages use set summary when available, otherwise a tune-count fallback.
  - Tune pages use summary when available, otherwise a fallback built from tune type, chart metadata, and set membership count.

## Testing

### How to Test

Automated:

- `npm run lint`
- `npm run test`
- `npm run typecheck`
- `npm run build`

Manual:

1. Open `/sets` and verify set titles link to `/sets/[slug]`.
2. Open a set detail page and verify the ordered tune list links to `/tunes/[slug]`.
3. Open a tune detail page and verify aliases, chart metadata, always-visible chart content, and linked set memberships.
4. Open `/tunes`, click a row to expand its chart, then click a tune or set link inside the row and verify navigation happens without toggling the row.
5. Visit an unknown `/sets/<missing>` or `/tunes/<missing>` URL and verify the shared not-found experience appears.

### Edge Cases

- **Set without summary**: The set detail page omits the summary block cleanly.
- **Tune without public set memberships**: The tune detail page shows a neutral empty-state message instead of an empty list.
- **Repeated tune usage across sets**: Tune pages surface all public set memberships with stable slugs.
- **Unknown or non-public slugs**: Repository lookups return `undefined`, and the route responds with `notFound()`.

## Limitations and Future Work

- Set entry badges still use the existing placeholder styling because set entries do not yet expose tune type.
- The homepage still shows legacy `nextIssue` roadmap labels for the Tunes and Sets cards; this branch updated the live-surface copy and status but did not redefine that planning metadata.
- Search still targets tunes only and does not yet surface set detail routes.
- Metadata is intentionally lightweight (title and description only); richer social metadata can be added later if needed.
